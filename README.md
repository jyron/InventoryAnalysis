# Retail Reconciliation — Take-Home Deliverable

Reconciles POS (CSV), inventory management (Excel), and e-commerce (JSON) exports into a single SKU-keyed view, produces an inventory health analysis, and uses an LLM (with Pydantic-constrained outputs) to triage free-text ops notes and draft a CFO summary.

## Quick start

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
jupyter notebook analysis.ipynb
```

The notebook executes end-to-end without an API key (offline fallbacks for the AI layer). To enable the Anthropic-backed LLM paths:

```bash
export ANTHROPIC_API_KEY=sk-ant-...
```

## Layout

```
retail_reconcile/       # Reusable package — this is the "for the next consultant" layer
├── loaders.py          # Source-specific loaders → canonical schemas
├── normalize.py        # SKU / date / product-name normalization (deterministic)
├── reconcile.py        # Cross-source matching, including fuzzy e-com→SKU
├── insights.py         # Stockout risk, dead inventory, reconciliation gaps, ...
├── ai_insights.py      # Pydantic schemas + LLM triage/summary + offline fallbacks
└── config.py           # Client-specific knobs — intentionally isolated

analysis.ipynb          # Walkthrough — executes the whole pipeline end-to-end
EXECUTIVE_SUMMARY.md    # 1-page CFO summary
PRESENTATION.md         # Speaker notes for the interview walkthrough
outputs/                # CSV/JSON artifacts generated by the notebook
```

## What's reusable vs. client-specific

| Component | Reusable? | Notes |
|---|---|---|
| `normalize.py` | Yes | Generic across retail clients. |
| `reconcile.py` | Yes | Works on any two-namespace product system. |
| `insights.py` | Yes | Thresholds come from `config.py`. |
| `ai_insights.py` | Yes | Pydantic schemas, no client facts hardcoded. |
| `config.py` | **Client-specific** | SKU prefixes, fuzzy threshold, priority order. |
| `loaders.load_inventory` column renames | **Partially** | Would become YAML in production. |
